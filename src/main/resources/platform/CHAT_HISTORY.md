### Что такое GraphQL?
GraphQL — это язык запросов для API, созданный Facebook в 2015 году. Это не база данных и не фреймворк — это спецификация того, как клиент общается с сервером.
Главная идея: клиент сам решает, какие данные получить, а не сервер диктует формат ответа.

### REST vs GraphQL

| | REST | GraphQL |
|---|---|---|
| Эндпоинтов | Много (`/employees`, `/departments`, `/orgs`) | Один (`/graphql`) |
| Формат запроса | URL + HTTP метод | Тело запроса (query/mutation) |
| Данные в ответе | Фиксированные (сервер решает) | Гибкие (клиент решает) |
| Over-fetching | Да (лишние поля приходят всегда) | Нет (только то что просишь) |
| Under-fetching | Да (N запросов для связанных данных) | Нет (всё за один запрос) |
| Версионирование | `/v1/`, `/v2/` | Не нужно (схема эволюционирует) |
| Документация | Swagger/OpenAPI отдельно | Схема = документация |

Пример: нужно показать сотрудников с отделом и организацией — в REST 3 запроса, в GraphQL один:
```graphql
query {
  employees {
    firstName
    department { name }
    organization { name }
  }
}
```

### GraphQL vs WebFlux
Это разные уровни — не надо противопоставлять. WebFlux — реактивный стек (Mono/Flux, неблокирующий I/O). GraphQL — протокол/формат запросов к API. Они могут работать вместе. В этом проекте WebFlux подключён только для тестов.

### Как устроен этот проект

```
Клиент → POST /graphql → Spring GraphQL → Controller (@QueryMapping/@MutationMapping)
```

3 сущности с иерархией: Organization → Department[] + Employee[]

### Почему .graphqls файлы в resources, а не в java?

В REST контракт API и логика находятся в одном месте (Java контроллер). В GraphQL они строго разделены:
```
src/main/resources/graphql/
    department.graphqls   ← КОНТРАКТ (что API умеет, какие типы)
    v2.graphqls
    v3.graphqls

src/main/java/
    controller/DepartmentController.java  ← ЛОГИКА (как это работает)
    repository/DepartmentRepository.java
    entities/Department.java
```
`.graphqls` — это текстовый файл-схема, не Java код. Spring при старте читает все `.graphqls` из `resources/graphql/` и строит из них граф. Это похоже на `application.yml` — конфигурация, не код. Если схема и Java код расходятся — Spring выдаст ошибку при старте, не в runtime.

### Три типа операций

| Операция | Аналог в REST | Для чего |
|---|---|---|
| `Query` | GET | Читать данные |
| `Mutation` | POST / PUT / DELETE | Изменять данные |
| `Subscription` | WebSocket | Подписка на события в реальном времени |

### Разбор department.graphqls построчно

```graphql
type Query {
   departments: [Department]         // GET /departments → список
   department(id: ID!): Department!  // GET /departments/{id} → один объект
}

type Mutation {
   newDepartment(department: DepartmentRequestDto!): Department    // POST
   updateDepartment(department: DepartmentRequestDto!, id: ID!): Department // PUT
   deleteDepartment(id: ID!): Department  // DELETE
}

input DepartmentRequestDto {
   name: String!        // обязательное поле (аналог @RequestBody в REST)
   organizationId: Int  // необязательное (null разрешён)
}

type Department {
   id: ID!                     // никогда не null
   name: String!               // никогда не null
   organization: Organization  // может быть null
   employees: [Employee]       // список, может быть null
}
```

### Связь схемы с Java кодом

```
department.graphqls               DepartmentController.java
─────────────────────────────────────────────────────────
type Query {                      @QueryMapping
  departments: [Department]  →    List<Department> departments()

  department(id: ID!)        →    @QueryMapping
                                  Department department(@Argument Integer id)
}

type Mutation {                   @MutationMapping
  newDepartment(...)         →    Department newDepartment(@Argument DepartmentRequestDto dto)
}
```
Схема = обязательство. Java код = выполнение этого обязательства.

### Почему три файла?
Организационно — разбили по сущностям. `extend type Query` означает "добавить методы в уже объявленный Query из другого файла".

### `type` vs `input` — в чём разница

В GraphQL есть строгое правило: объекты для чтения и объекты для записи — разные вещи.

| | `type` | `input` |
|---|---|---|
| Направление | Сервер → Клиент | Клиент → Сервер |
| Аналог в REST/Java | `@ResponseBody` / Entity | `@RequestBody` / DTO |
| Может содержать | Поля + вложенные объекты | Только скалярные поля и другие input типы |
| Используется в | Возвращаемое значение | Аргумент метода |

`input` принимает только скалярные поля и другие input типы. `type` может содержать вложенные объекты, резолверы, циклические ссылки — поэтому его нельзя передавать как аргумент в мутацию.

```graphql
// Клиент ОТПРАВЛЯЕТ (для создания/обновления):
input DepartmentRequestDto {
   name: String!
   organizationId: Int   // просто число — только ID
}

// Сервер ВОЗВРАЩАЕТ (результат):
type Department {
   id: ID!
   name: String!
   organization: Organization  // полный объект со всеми полями
   employees: [Employee]       // список объектов
}
```

Обрати внимание: клиент при создании знает только `organizationId: Int`. Когда получает данные обратно — видит полный объект `organization: Organization` с именем и всеми полями.

Аналогия с Java:
```java
@PostMapping("/departments")
public DepartmentResponseDto create(@RequestBody DepartmentRequestDto dto) {
    // dto    = input в GraphQL (что пришло от клиента)
    // return = type  в GraphQL (что отдаём клиенту)
}
```

Почему для `deleteDepartment` не создали `input DeleteDepartmentDto { id: ID! }`, а передают `id: ID!` напрямую? Когда имеет смысл создавать `input` объект, а когда достаточно скалярного значения?

**Вопрос:**
`department(id: ID!): Department!` возвращает с `!`, а `newDepartment(...): Department` без `!`. Я не знаю точно, но думаю что при создании возвращаемый объект может быть null — такая логика у Mutation.

**Ответ:**

Ты мыслишь в правильном направлении. `!` в возвращаемом типе — это гарантия результата.

### Query — жёсткая гарантия (`Department!`)
Когда клиент запрашивает `department(id: 5)` — он ожидает найти конкретную запись. Если записи нет — это исключительная ситуация. Сервер должен либо вернуть объект, либо бросить ошибку. Никогда не вернёт null.
```
Клиент: "Дай мне департамент с id=5"
Сервер: либо → { id: 5, name: "IT" }
        либо → ERROR: "Department not found"
        никогда → null
```

### Mutation — мягкий отказ (`Department` без `!`)
При создании могут быть ситуации когда операция "не удалась" но не является критической ошибкой:
```
Клиент: "Создай департамент"
Сервер: либо → { id: 6, name: "HR" }   // успех
        либо → null                     // что-то пошло не так, но не exception
```

### Важный нюанс — это проектное решение
Многие GraphQL API делают мутации тоже с `!`, используя `errors` в ответе. Оба подхода правильные.

### Правило большого пальца

| Ситуация | Возвращаемый тип | Логика |
|---|---|---|
| Запрос по ID | `Type!` | Запись должна существовать |
| Запрос списка | `[Type]` | Пустой список — не ошибка |
| Создание/обновление | `Type` или `Type!` | Зависит от стратегии ошибок |
| Удаление | `Type` | Возвращает удалённый объект или null |

### Как читать .graphqls файлы

`.graphqls` — это контракт API. Читай его как меню ресторана: написано что можно заказать, с какими параметрами и что получишь в ответ.

**Грамматика чтения — три вопроса:**
```graphql
department(id: ID!): Department!
   ↑            ↑        ↑
"метод"    "что принять"  "что вернуть"
```
Читай вслух: "Метод `department`, принимает `id` типа ID (обязательный), возвращает объект `Department` (гарантированно не null)"

**Разбор трёх файлов проекта:**
```graphql
── department.graphqls ──────────────────────────────────
type Query {
  departments: [Department]          // "дай все департаменты" → список
  department(id: ID!): Department!   // "дай один по id" → объект
}

── v3.graphqls ──────────────────────────────────────────
extend type Query {         // "extend" = дополняем Query из другого файла
  organizations: [Organization]
  organization(id: ID!): Organization!
  organizationName(name: String!): Organization!  // поиск по имени
}

── v2.graphqls ──────────────────────────────────────────
extend type Query {
  employees: [Employee]
  employeesWithFilter(filter: EmployeeFilter): [Employee]  // с фильтрацией
  employee(id: ID!): Employee!
}
input EmployeeFilter {
  salary: FilterField    // фильтр по зарплате
  age: FilterField       // фильтр по возрасту
  position: FilterField  // фильтр по должности
}
input FilterField {
  operator: String!  // "gt" (больше), "lt" (меньше), "eq" (равно)
  value: String!     // "50000"
}
```

**Почему три файла, а не один?**
Организационно — разбили по сущностям. Spring читает все `.graphqls` из папки и склеивает в одну схему. `extend type Query` — добавить методы в уже объявленный Query.

### .graphqls усложняет ли логику? Честное сравнение

| | REST | GraphQL |
|---|---|---|
| Контракт | Нет (или Swagger отдельно) | Обязательный `.graphqls` |
| Добавить метод | Написал контроллер — готово | Схема + контроллер |
| Документация | Swagger (поддерживать отдельно) | Схема = документация (всегда актуальна) |
| Ошибка несоответствия | В runtime | При старте приложения |

Да, на старте чуть больше работы. Но это окупается — схема всегда актуальна и является живой документацией.

### Откуда фронтенд знает какие запросы можно делать?

Это называется **интроспекция** — ключевая особенность GraphQL. Сервер всегда отвечает на запросы о своей схеме. В GraphiQL нажми кнопку **"Docs"** (справа) — там все доступные операции, типы и описания. Это живая документация.

Инструменты для фронтенда:
- **Apollo Client** — генерирует TypeScript типы из схемы
- **GraphQL Code Generator** — генерирует код под любой язык

### Когда применять GraphQL vs REST

**GraphQL выигрывает:**
- Сложные связанные данные (как этот проект: Organization → Departments → Employees)
- Несколько клиентов (web, mobile, TV) — каждый берёт только нужные поля
- Быстро меняющийся продукт — добавляешь поля без версионирования
- Мобильное приложение — меньше данных по сети

**REST выигрывает:**
- Простой CRUD без сложных связей
- Публичный API для внешних разработчиков — понятнее и лучше кешируется
- Файлы, бинарные данные — нативная поддержка
- Команда не знает GraphQL

### Что быстрее на продакшене?

| Сценарий | Победитель | Почему |
|---|---|---|
| Один простой запрос | REST | Меньше overhead на парсинг |
| Несколько связанных запросов | GraphQL | 1 запрос вместо 3-5 |
| Мобильный клиент | GraphQL | Меньше данных по сети |
| Кеширование | REST | HTTP кеш работает нативно |

**Главная ловушка GraphQL — N+1 проблема:**
```
Запрос: organizations → departments → employees
Наивная реализация:
  1 запрос на organizations
+ 3 запроса на departments (по одному на каждую org)
+ 15 запросов на employees (по одному на каждый dept)
= 19 запросов в БД вместо 1!
```
В этом проекте решено через `@BatchMapping` — разберём в Блоке 2.

### Две эпохи GraphQL в Spring

**Старый подход — `com.graphql-java` (2017–2021):**
Community-библиотеки, появились когда Spring ещё не имел официальной поддержки GraphQL. Версия 5.x — это 2019 год. Использовали другую модель: интерфейсы-резолверы `GraphQLQueryResolver`, `GraphQLMutationResolver`.

```java
// СТАРЫЙ стиль
@Component
public class DepartmentResolver implements GraphQLQueryResolver {
    public List<Department> departments() { ... }
}
```

**Новый подход — `spring-boot-starter-graphql` (с 2022):**
Официальная интеграция от Spring. Внутри уже содержит `graphql-java` (движок). Использует аннотации `@QueryMapping`, `@MutationMapping` — именно так в твоём проекте.

```java
// НОВЫЙ стиль — именно это в твоём проекте
@Controller
public class DepartmentController {
    @QueryMapping
    public List<Department> departments() { ... }
    @QueryMapping
    public Department department(@Argument Integer id) { ... }
}
```

Конфликт — оба стартера попытаются зарегистрировать GraphQL сервлет на /graphql. Приложение не запустится или поведение станет непредсказуемым.

**Зависимости в pom.xml — полная картина:**
- `spring-boot-starter-graphql` — основа: GraphQL API, аннотации, схема
- `spring-graphql-test` — тестирование GraphQL запросов
- `spring-boot-starter-web` — HTTP сервер (GraphQL работает поверх HTTP)
- `spring-boot-starter-data-jpa` — JPA + Hibernate
- `postgresql` — драйвер БД
- `spring-webflux` (test scope) — нужен для тестирования через WebClient
- `lombok` — генерация геттеров/сеттеров

Всё что нужно для GraphQL — уже есть. Если видишь в интернете примеры с `GraphQLQueryResolver` — это устаревший код 2018-2020 годов, не применяй к своему проекту.

**Вопрос:**
GraphQL и REST – два разных подхода к разработке API для обмена данными через Интернет. REST дает возможность клиентским приложениям обмениваться данными с сервером с помощью команд HTTP – стандартного протокола связи в Интернете. А GraphQL – это язык запросов API, определяющий спецификации, по которым клиентское приложение должно запрашивать данные с удаленного сервера. Вы можете использовать GraphQL в вызовах API для определения запроса, не полагаясь на серверное приложение. GraphQL и REST – две мощные технологии, на которых основывается большинство наших современных приложений.

GET, POST, PUT, DELETE, PATCH — объясни в чем разница?

**Ответ:**

Когда мы проектируем или используем REST API, эти пять методов определяют тип действия, которое мы хотим совершить с ресурсом (например, пользователем, заказом или статьей).

### 1. GET — Получение данных

Используется исключительно для чтения информации. Он не должен изменять состояние сервера.

- **Идемпотентность:** Да (сколько бы раз вы ни вызывали GET, данные на сервере не изменятся)
- **Безопасность:** Да
- **Пример:** Получить список всех товаров в магазине

### 2. POST — Создание данных

Используется для создания нового ресурса. Данные передаются в теле запроса.

- **Идемпотентность:** Нет (если нажать «Отправить» дважды, в базе могут создаться две одинаковые записи, например, два заказа)
- **Безопасность:** Нет
- **Пример:** Регистрация нового пользователя или создание нового поста в блоге

### 3. PUT — Полное обновление

Используется для замены существующего ресурса целиком. Если ресурс не найден, PUT может создать его (зависит от реализации).

- **Идемпотентность:** Да (если вы отправите один и тот же объект 10 раз, результат будет одинаковым — объект просто перезапишется тем же самым)
- **Важно:** Нужно отправлять все поля объекта. Если вы отправите только имя, остальные поля (например, возраст или email) могут быть стёрты или заменены на значения по умолчанию
- **Пример:** Обновление профиля пользователя, где вы присылаете весь объект целиком

### 4. PATCH — Частичное обновление

Используется для изменения части ресурса.

- **Идемпотентность:** Обычно нет (но зависит от реализации)
- **Пример:** Вы хотите изменить только email пользователя, не затрагивая его имя, фамилию и дату рождения. Вы отправляете только это поле

### 5. DELETE — Удаление

Удаляет указанный ресурс.

- **Идемпотентность:** Да (первый вызов удалит ресурс, последующие не изменят состояние сервера, так как ресурса уже нет)
- **Пример:** Удаление фотографии из профиля

### Сводная таблица

| Метод  | Действие           | Идемпотентность | Тело запроса |
|--------|--------------------|-----------------|--------------|
| GET    | Чтение             | Да              | Нет          |
| POST   | Создание           | Нет             | Да           |
| PUT    | Полная замена      | Да              | Да           |
| PATCH  | Частичное изменение| Нет             | Да           |
| DELETE | Удаление           | Да              | Нет          |

### Что такое идемпотентность?

Это важное понятие в архитектуре API. Метод называется идемпотентным, если повторный запрос к серверу с тем же набором данных приводит к тому же результату, что и первый успешный запрос. Это гарантирует стабильность системы при сетевых сбоях (например, если клиент не получил подтверждение и отправил запрос повторно).

---

**Вопрос:**
Повторный запрос к серверу с тем же набором данных приводит к тому же результату — тогда как DELETE удаление того же результата не будет, один раз удалили и всё. Почему идемпотентным считается?

**Ответ:**

Это отличный вопрос, который часто сбивает с толку. Разгадка кроется в том, что именно мы считаем «результатом».

В контексте идемпотентности **результат — это состояние ресурсов на сервере**, а не ответ сервера (код состояния HTTP).

### Почему DELETE идемпотентен?

Представьте ситуацию:

**Первый запрос** `DELETE /users/42`:
- Сервер удаляет пользователя с ID 42
- Состояние сервера: пользователя 42 больше нет
- Ответ: `200 OK` или `204 No Content`

**Второй запрос** `DELETE /users/42`:
- Сервер видит, что пользователя 42 нет
- Состояние сервера: пользователя 42 по-прежнему нет — оно не изменилось по сравнению с состоянием после первого запроса
- Ответ: `404 Not Found`

Несмотря на то что ответы разные (204 в первый раз и 404 во второй), **конечное состояние базы данных одинаково** — записи нет. Именно поэтому метод считается идемпотентным.

### Сравнение с неидемпотентным POST

- Отправляете `POST /orders` → Сервер создаёт заказ №101
- Отправляете тот же `POST /orders` ещё раз → Сервер создаёт ещё один заказ — №102

После второго вызова состояние сервера изменилось: теперь там два заказа вместо одного. Это и есть отсутствие идемпотентности.

### Коротко о главном

Идемпотентность — это про то, **изменится ли мир (база данных) после второго, третьего или десятого идентичного запроса**.

Для DELETE, PUT и GET мир меняется только один раз (или вообще не меняется), сколько бы раз вы ни дублировали запрос. Поэтому если из-за плохого интернета клиент отправил запрос на удаление дважды — это безопасно, вы просто удалите то, что уже удалено, и не повредите другие данные.

**Вопрос:**
GraphQL как то связан с HTTP?

**Ответ:**

Да, GraphQL напрямую связан с HTTP и в подавляющем большинстве случаев работает именно поверх этого протокола.

Хотя технически GraphQL является «протоколо-независимым» (его можно передавать через WebSocket, gRPC или даже через другие протоколы), на практике HTTP — это стандарт де-факто для передачи GraphQL-запросов.

### Как GraphQL использует HTTP

В отличие от REST, где используются разные методы (GET, POST, PUT, DELETE) и множество разных URL, GraphQL упрощает схему:

- **Один Endpoint:** Обычно это один URL (например, `/graphql`)
- **Один метод — POST:** Почти все запросы в GraphQL отправляются методом POST. Даже если вы просто хотите «прочитать» данные (что в REST делается через GET), вы отправляете JSON-тело с описанием запроса внутри POST-запроса

> Примечание: Иногда для простых Query-запросов используют GET с параметрами в URL, но это встречается реже из-за ограничений по длине строки.

### Структура запроса

В REST HTTP-метод определяет действие. В GraphQL действие (чтение, запись или подписка) определяется **внутри тела запроса**, а не средствами HTTP.

- **REST:** `GET /users/1`
- **GraphQL:** `POST /graphql` с телом `query { user(id: 1) { name } }`

### Обработка статус-кодов HTTP

В REST мы сильно полагаемся на статус-коды: 201 (создано), 404 (не найдено), 500 (ошибка сервера).

В GraphQL сервер **почти всегда возвращает статус-код 200 OK**, даже если в самом запросе произошла ошибка. Информация об ошибках передается внутри JSON-ответа в специальном массиве `"errors"`.

### Главные отличия в контексте HTTP

| Характеристика | REST                                      | GraphQL                             |
|----------------|-------------------------------------------|-------------------------------------|
| Количество URL | Много (`/users`, `/posts`, `/orders`)     | Один (`/graphql`)                   |
| Методы HTTP    | Использует весь спектр (GET, POST и т.д.) | В основном только POST              |
| Кэширование    | На уровне HTTP (браузером, CDN)           | На уровне клиента (Apollo, Relay)   |
| Статус-коды    | Отражают результат операции               | Почти всегда 200 OK                 |

**Итог:** GraphQL не заменяет HTTP, он использует его как «транспорт». Если REST относится к HTTP как к набору команд, то GraphQL относится к HTTP как к простому туннелю, по которому передаётся его собственный сложный язык запросов.

**Вопрос:**
В GraphQL что такое Graph и что такое QL?

**Ответ:**

Название GraphQL расшифровывается как **Graph Query Language** (язык запросов к графам).

### 1. Что такое Graph (Граф)?

В контексте GraphQL ваши данные представляются не в виде плоских таблиц (как в Excel), а в виде **графа** — сети связанных между собой объектов.

- **Узлы (Nodes):** Это объекты (например, «Пользователь», «Пост», «Комментарий»)
- **Рёбра (Edges):** Это связи между ними (например, пользователь «написал» пост, пост «содержит» комментарии)

В REST API данные часто разрознены: чтобы получить профиль пользователя и его последние 10 постов, нужно сделать два разных запроса. В GraphQL вы просто «идёте» по графу от одного узла к другому за один запрос.

**Аналогия:** Представьте социальную сеть. Вы — точка в центре. От вас идут линии к вашим друзьям, от друзей — к их интересам. Это и есть граф. GraphQL позволяет сказать: «Начни с меня, перейди к моим друзьям, а от них — к их фотографиям».

### 2. Что такое QL (Query Language)?

**Query Language** — это «язык запросов». Точно так же, как SQL используется для общения с базами данных, GraphQL используется для общения с вашим API.

Ключевая особенность: **клиент сам выбирает, что ему нужно**.

```graphql
query {
  user(id: "123") {
    name           # Хочу только имя
    friends {      # Имена его друзей
      name
    }
  }
}
```

Сервер посмотрит на этот «запрос» и вернёт JSON ровно такой же структуры. Никаких лишних данных.

### Почему это работает вместе?

Смысл GraphQL в том, чтобы дать мощный язык (**QL**) для навигации по огромной сети связанных данных (**Graph**). Вы не думаете о «эндпоинтах» или «таблицах». Вы просто берёте «карту» ваших данных и с помощью специального синтаксиса указываете путь, по которому нужно пройти, чтобы собрать нужную информацию за один заход.

---

**Вопрос:**
В чём же преимущество GraphQL если база данных Postgres и надо достать пользователя, посты и комментарии — всё равно же будет три запроса разных к базе данных. Зачем нам тогда GraphQL если можно применять REST?

**Ответ:**

Это отличный вопрос, который бьёт прямо в «больную мозоль» GraphQL — проблему N+1.

Вы абсолютно правы: если просто написать код «в лоб», то для получения пользователя, его 10 постов и комментариев к ним, GraphQL-сервер может сделать 1 запрос к пользователю, 1 к постам и 10 к комментариям (всего 12!). Это будет даже хуже, чем в REST.

Но преимущество GraphQL раскрывается на двух уровнях: **клиентском** и **инженерном**.

### 1. Со стороны фронтенда (Клиент)

- **В REST:** Нужно написать код для трёх вызовов по сети. Нужно обработать три ожидания (loading), три возможные ошибки и склеить эти данные на клиенте. Это тратит заряд батареи телефона и время пользователя (лишние «рукопожатия» HTTP).
- **В GraphQL:** Делается один сетевой запрос. Весь хаос со сбором данных переносится на сервер, где связь с базой данных в сотни раз быстрее, чем связь телефона с вышкой 4G.

### 2. Со стороны бэкенда — решение проблемы запросов (DataLoader)

Чтобы GraphQL не «засыпал» Postgres кучей мелких запросов, используются специальные инструменты — **DataLoader**.

Вместо того чтобы выполнять запросы сразу, DataLoader «собирает» все ID, которые нужно достать, и объединяет их в один запрос:
```sql
SELECT * FROM comments WHERE post_id IN (1, 2, 3...)
```
В итоге к базе данных всё равно уходит минимум запросов (часто те же 2-3), но клиент получает всё «одним пакетом».

### 3. Главные преимущества (почему не REST?)

| Ситуация        | REST                                                               | GraphQL                                           |
|-----------------|--------------------------------------------------------------------|----------------------------------------------------|
| Overfetching    | Сервер отдаёт все поля профиля (даже если нужно только имя). Лишний трафик. | Клиент запрашивает только `name`. Экономия трафика. |
| Underfetching   | Одного запроса мало, нужна цепочка user → posts → comments.       | Один запрос забирает всё дерево данных сразу.      |
| Версионность    | При изменении структуры нужна `/v2/`, `/v3/`.                     | API эволюционирует плавно. Просто добавляются новые поля. |
| Типизация       | Нужна документация (Swagger), которая может врать.                | Строгая схема (Schema). Клиент всегда знает, какие данные можно получить. |

### Резюме

GraphQL не заменяет SQL и не делает магию внутри базы данных. Его главная цель — **отвязать фронтенд от структуры бэкенда**.

- Если у вас простое CRUD-приложение, где вы всегда выводите одну таблицу — **REST проще и лучше**.
- Если у вас сложный интерфейс (как лента Facebook или Dashboard Jira), где в разных местах нужны разные кусочки связанных данных — **GraphQL экономит месяцы разработки** и делает работу приложения быстрее для конечного пользователя.

**Вопрос:**
Если в проекте GraphQL и микросервисная архитектура, то по-особенному гейтвей надо будет настроить?

**Ответ:**

Да, в микросервисной архитектуре с GraphQL гейтвей настраивается принципиально иначе, чем в REST.

### REST + микросервисы (обычный API Gateway)

```
Client → API Gateway (nginx/Kong) → /users-service
                                  → /orders-service
                                  → /products-service
```
Гейтвей просто **проксирует** запросы на нужный сервис по URL.

### GraphQL + микросервисы — два подхода

**Подход 1: Schema Stitching (устаревший)**

Гейтвей собирает схемы от каждого сервиса вручную. Сложно, хрупко, почти не используется.

**Подход 2: Apollo Federation (современный стандарт)**

```
Client → Apollo Router (Gateway)
              ↓
    ┌─────────┬──────────┬──────────┐
    │ Users   │ Orders   │ Products │
    │ subgraph│ subgraph │ subgraph │
    └─────────┴──────────┴──────────┘
```

Каждый микросервис — это **subgraph** со своей частью схемы. Apollo Router сам собирает их в единый граф и знает, к какому сервису идти за какими данными.

Ключевая особенность — сервисы могут **расширять типы друг друга**:

```graphql
# users-service
type User @key(fields: "id") {
  id: ID!
  name: String!
}

# orders-service — расширяет User из другого сервиса
extend type User @key(fields: "id") {
  orders: [Order]  # orders-service знает заказы пользователя
}
```

Клиент делает один запрос, Router сам идёт в нужные сервисы.

### Для Spring Boot

```xml
<!-- каждый микросервис остаётся на spring-boot-starter-graphql -->
<!-- добавляется только поддержка федерации -->
<dependency>
    <groupId>com.apollographql.federation</groupId>
    <artifactId>federation-graphql-java-support</artifactId>
</dependency>
```

### Сравнение подходов

| | REST Gateway | GraphQL Federation |
|---|---|---|
| Роутинг | По URL | По типам и полям схемы |
| Объединение данных | На клиенте или BFF | На уровне Router автоматически |
| Документация | Swagger per service | Единая схема всего графа |

В текущем учебном проекте это не нужно — он монолит. Федерация актуальна когда реально несколько отдельных сервисов с разными командами.

### Один эндпоинт

Все запросы идут на **один** `POST /graphql`. Контроллеры — это не эндпоинты, это внутренние обработчики. Spring GraphQL смотрит внутрь тела запроса и вызывает нужный метод.

```
REST:   GET /departments, GET /employees, POST /departments ... — много URL
GraphQL: POST /graphql (всегда один URL, Spring роутит внутри)
```

### Что такое резолвер

Схема (`.graphqls`) — это объявление того, что можно запросить. Резолвер — это Java-метод который фактически выполняет этот запрос.

```graphql
# Схема — объявление (НЕ резолвер)
type Query {
   departments: [Department]
}
```
```java
// Резолвер — Java метод с аннотацией
@QueryMapping
public List<Department> departments() {  // имя метода = имя в схеме
    return repository.findAll();
}
```

В других фреймворках это явно называется `resolver`. В Spring GraphQL — аннотации `@QueryMapping`, `@MutationMapping`, `@SchemaMapping`. Связь по имени метода — ты правильно ответил.

### HTTP и хедеры

Да, GraphQL работает поверх HTTP. JWT токен передаётся в хедере:

```http
POST /graphql
Authorization: Bearer eyJhbGci...
Content-Type: application/json
```

Поймать в Spring:
```java
@QueryMapping
public List<Department> departments(
        @RequestHeader("Authorization") String token) { ... }
```

Всё что работало в REST — хедеры, куки, HTTPS, Spring Security — работает и в GraphQL.

### Конкретное преимущество — живой пример

Карточка пользователя: имя + последние 3 поста + лайки на каждом.

**REST:** 5 запросов по сети, лишние поля (email, phone, bio) приходят но не нужны.

**GraphQL:**
```graphql
query {
  user(id: 42) {
    name
    posts(last: 3) {
      title
      likesCount
    }
  }
}
```
1 запрос, только нужные поля.

**Вопрос:**
"Клиент видит единую схему" — это как?

```graphql
}
```

Инструменты типа **Apollo Client** делают интроспекцию при старте и автоматически генерируют TypeScript типы для фронтенда. Фронтендер никогда не смотрит в `.graphqls` файлы напрямую.

**Вопрос:**
Разруливает ли запросы с фронтенда DispatcherServlet или он тут ни при чём?

**Ответ:**

DispatcherServlet участвует, но частично.

```
POST /graphql
     │
     ▼
DispatcherServlet       ← принимает HTTP запрос, передаёт GraphQlHttpHandler
     ▼
GraphQlHttpHandler      ← парсит тело запроса, читает имя операции
     ▼
Spring GraphQL Router   ← ищет метод с нужной аннотацией
     ▼
@QueryMapping departments()  ← вызывает нужный метод
```

DispatcherServlet участвует только на HTTP уровне — принимает `POST /graphql` и передаёт специальному `GraphQlHttpHandler`. Маршрутизацией между `departments()`, `employees()` и другими методами занимается уже Spring GraphQL по содержимому тела запроса, а не DispatcherServlet.

В REST DispatcherServlet маршрутизирует по URL + HTTP метод. В GraphQL — только пропускает запрос внутрь, дальше GraphQL сам разруливает.

Нет. В GraphQL два разных языка:

**SDL (Schema Definition Language)** — это `.graphqls` файлы. Пишет бэкендер. Описывает что сервер умеет.
```graphql
type Query {
  departments: [Department]
}
```

```graphql
query {
  departments {
    id
    name
  }
}
```

| | Кто пишет | Назначение |
|---|---|---|
| SDL | Бэкендер | Описывает что API умеет |
| Query Language | Клиент | Указывает что клиент хочет получить |

Книга говорит о синтаксисе запросов (`query {}`, `mutation {}`, `subscription {}`) который клиент отправляет на сервер. `.graphqls` — это схема сервера, не то о чём говорит эта цитата.

**Вопрос / Уточнение студента:**

В GraphQL смешиваются сразу несколько понятий:

**1. GraphQL Query Language — язык запросов клиента (НЕ `.graphqls`)**
```graphql
query {
  books {
    id
    title
  }
}
```
Клиент говорит: "Я хочу books, и у каждой книги только id и title" — это и есть "client specifies intent". Клиент сам описывает что получить, какие поля нужны, какую mutation выполнить.

**2. `.graphqls` — это Schema**
```graphql
type Query {
  books: [Book]
}
type Book {
  id: ID
  title: String
}
```
Описание API: какие Query существуют, какие Mutation, какие типы данных. Это контракт API.

**3. Java backend — выполняет запрос**
```java
@QueryMapping
public List<Book> books() {
    return bookService.findAll();
}
```
GraphQL читает schema → понимает что `books` существует → вызывает resolver (`@QueryMapping`) → возвращает только запрошенные поля.

**Самое важное понимание:**

В REST — **сервер** решает структуру ответа:
```
GET /books/1  →  { "id": 1, "title": "Java", "description": "...", "authors": [...] }
```

В GraphQL — **клиент** решает структуру ответа:
```graphql
query {
  book(id: 1) {
    title       ← только это поле
  }
}
```
```json
{ "data": { "book": { "title": "Java" } } }
```

Вот в этом и главная идея GraphQL.

Метод в контроллере `myPets` возвращает и имя и цвет, но в запросе цвет не нужен — это ошибка бэкенда, избыточность?

**Ответ:**

Нет — это и есть главная идея GraphQL. Бэкенд возвращает объект целиком (`name + color`), но GraphQL Engine обрезает ответ до того что запросил клиент. По сети лишние данные не гоняются.

```graphql
query { pets { name } }  ← клиент просит только name
```

```json
{ "data": { "pets": [{ "name": "Luna" }] } }  ← color не пришёл
```

В REST сервер всегда отдаёт фиксированный набор полей. В GraphQL клиент сам выбирает нужные поля — это называется **no overfetching**.

**Вопрос:**
Ты ничего не объяснил про аннотации `@QueryMapping` и другие, а уже спрашиваешь про `@SchemaMapping`. Давай сначала дашь теорию.

**Ответ:**

### @QueryMapping

```java
@QueryMapping
public Iterable<Department> departments() { ... }
```

Говорит Spring: "этот метод обрабатывает поле `departments` из `type Query` в схеме". Связь по имени метода. Аналог в REST: `@GetMapping("/departments")`.

### @MutationMapping

```java
@MutationMapping
public Department newDepartment(@Argument DepartmentRequestDto department) { ... }
```

То же что `@QueryMapping`, но для `type Mutation`. Аналог: `@PostMapping`.

### @Argument

```java
public Department department(@Argument Integer id) { ... }
```

Извлекает аргумент из GraphQL-запроса и подставляет в параметр метода. Аналог `@PathVariable` / `@RequestBody` из REST.

### @SchemaMapping

```java
@SchemaMapping(typeName = "Department", field = "organization")
public Organization organization(Department department) {
    return department.getOrganization();
}
```

Нужен для вложенных полей. Когда клиент запрашивает `departments { organization { name } }`, Spring не знает как достать `organization` для каждого `Department` — для этого и нужен `@SchemaMapping`. Параметр метода — текущий объект (тот Department для которого нужна organization). Вызывается отдельно для каждого объекта → **N+1 проблема**.

### @BatchMapping — решение N+1

```java
@BatchMapping(typeName = "Department", field = "employees")
public Map<Department, Set<Employee>> employees(List<Department> departments) {
    return departments.stream()
            .collect(Collectors.toMap(dep -> dep, Department::getEmployees));
}
```

Вместо вызова для каждого объекта по отдельности — вызывается **один раз** для всего списка. Возвращает `Map<Department, Set<Employee>>`, Spring сам раскладывает данные.

| | @SchemaMapping | @BatchMapping |
|---|---|---|
| Параметр | один объект | List всех объектов |
| Возвращает | один результат | Map результатов |
| Запросов к БД | N+1 | 1 |

**Вопрос:**
Что такое Scalar? Объясни.

**Ответ:**

### Scalar (Скалярный тип)

**Scalar** — это примитивный тип данных в GraphQL-схеме. Скаляры — это "листья" графа, у них нет вложенных полей.

### Встроенные скаляры

| GraphQL Scalar | Java аналог | Пример значения |
|---|---|---|
| `String` | `String` | `"John"` |
| `Int` | `Integer` / `int` | `42` |
| `Float` | `Double` / `float` | `3.14` |
| `Boolean` | `Boolean` / `boolean` | `true` / `false` |
| `ID` | `String` / `Long` | `"1"`, `"abc-123"` |

`ID` — особый тип. Семантически означает "уникальный идентификатор". По сети всегда передаётся как строка, но в Java может быть `Long`, `Integer` или `String` — Spring конвертирует.

### Где скаляры используются в проекте

```graphql
type Department {
   id: ID!          ← scalar ID
   name: String!    ← scalar String
}

input DepartmentRequestDto {
   name: String!         ← scalar String
   organizationId: Int   ← scalar Int
}
```

### Custom Scalar — зачем нужен?

Встроенных пяти скаляров часто не хватает. Например: `Date`, `DateTime`, `Email`, `URL`, `BigDecimal`.

Объявление в схеме:
```graphql
scalar Date
scalar BigDecimal

type Employee {
   salary: BigDecimal
   birthDate: Date
}
```

В Java регистрируется через библиотеку `graphql-scalars`:
```java
@Configuration
public class GraphQLConfig {
    @Bean
    public RuntimeWiringConfigurer runtimeWiringConfigurer() {
        return wiringBuilder -> wiringBuilder
            .scalar(ExtendedScalars.Date)
            .scalar(ExtendedScalars.GraphQLBigDecimal);
    }
}
```

### @specifiedBy

Необязательная директива для кастомных скаляров — ссылка на спецификацию формата:
```graphql
scalar DateTime @specifiedBy(url: "https://scalars.graphql.org/andimarek/date-time")
```

### Главное правило

Скаляр = тип без вложенных полей. Нельзя "войти внутрь" скаляра:
```graphql
# ПРАВИЛЬНО
query { departments { name id } }

# ОШИБКА — нельзя войти внутрь String
query { departments { name { length } } }
```

**Вопрос:**
Я забросил GraphQL на месяц, всё забыл. Начни объяснять с самого начала: что это за файл `pets.graphqls` и что означает:
```graphql
type Query {
    pets: [Pet]
    myPet(id: ID!): Pet
}

type Pet {
    name: String
    color: String
}
```

**Ответ:**

Файл `.graphqls` — это **контракт API**. Он описывает что клиент может запросить и как выглядят данные. Состоит из двух частей:

### Часть 1 — что можно заказать (операции):

```graphql
type Query {
    pets: [Pet]
    myPet(id: ID!): Pet
}
```

`type Query` — список всего что клиент может прочитать. Аналог GET-методов в REST.

| Строка | Читаем вслух | Аналог в REST |
|---|---|---|
| `pets: [Pet]` | "дай всех питомцев" → список | `GET /pets` |
| `myPet(id: ID!): Pet` | "дай одного питомца по id" | `GET /pets/{id}` |

- `[Pet]` — квадратные скобки = список объектов Pet
- `ID!` — восклицательный знак = обязательный параметр, без него запрос не выполнится
- `Pet` без скобок — один объект, не список

### Часть 2 — как выглядит объект:

```graphql
type Pet {
    name: String
    color: String
}
```

Описание структуры питомца. Оба поля без `!` — могут быть null. Это то что сервер возвращает клиенту.

### Главная идея — клиент сам выбирает что получить:

```graphql
# Только имена, без цвета
query {
    pets {
        name
    }
}
```

Сервер достанет оба поля из базы, но GraphQL обрежет ответ — по сети придёт только `name`. Это называется **no overfetching**.

**Вопрос:**

**Ответ:**

Нет. У питомца бизнес-логически должно быть имя. Правильнее:

```graphql
type Pet {
```

**Вопрос:**
Что будет если вызвать `myPet(id: "5")`? В списке только 2 питомца.

**Ответ:**

Java бросит `ArrayIndexOutOfBoundsException`. GraphQL поймает и вернёт:
```json
{
  "errors": [{ "message": "INTERNAL_ERROR", "path": ["myPet"] }],
  "data": { "myPet": null }
}
```

HTTP статус — **200**. Это особенность GraphQL: ошибки всегда внутри JSON, не в HTTP коде. Поле возвращает null потому что `myPet(id: ID!): Pet` — возвращаемый тип без `!`, то есть null разрешён.

**Вопрос:**
Что означает `(id: ID!)`? Может ли быть `myPet(id!: ID!)`?

**Ответ:**

`(id: ID!)` — это аргумент метода. `!` ставится только на **тип**, а не на имя:

```graphql
myPet(id: ID!)   ← ПРАВИЛЬНО
myPet(id!: ID!)  ← СИНТАКСИЧЕСКАЯ ОШИБКА
```

Аналогия в Java:
```java
public Pet myPet(String id)   // правильно
public Pet myPet(String! id)  // так не бывает
```

`ID!` — это отдельный тип "ID который никогда не null". Если передать null — GraphQL вернёт `ValidationError` до вызова Java кода.

**Вопрос:**
Чем отличается `pets: [Pet]` от `pets: [Pet!]!`?

**Ответ:**

Четыре варианта:
```graphql
pets: [Pet]    # список может быть null, элементы могут быть null
pets: [Pet!]   # список может быть null, но элементы не null
pets: [Pet]!   # список не null, но элементы могут быть null
pets: [Pet!]!  # список не null, и элементы не null
```

На практике чаще всего используют `[Pet!]!` — клиент гарантированно получит список (пусть пустой `[]`), а не null.

---

**Вопрос:**
`pets: [Pet!]!` — бекенд вернёт один элемент null, что тогда?

**Ответ:**

GraphQL уничтожит весь ответ через механизм **null propagation**:

- Элемент `Pet!` — null → ошибка, поднимается к списку
- Список `[Pet!]!` — тоже non-null → поднимается выше
- Nullable предка нет → `data: null`

Результат:
```json
{
  "errors": [{ "message": "..." }],
  "data": null
}
```

Компромисс на практике: `[Pet]!` — список точно придёт, но элемент может быть null. Тогда только один элемент будет null, остальные данные целые.

Правило: чем больше `!` — тем строже контракт, но тем опаснее если что-то пойдёт не так.
